# 误判陷阱：哪些情况不能作为否决理由

本指南面向 `unit_reviewer`（发现阶段）和 `challenger`（对抗复核阶段）。  
找到以下任意一种情形，必须选 `NEEDS_POC` 或保留 finding，**不能 REFUTED**。

**核心原则**：漏掉一个真实高危漏洞，比留下一个需要人工复查的误报代价高得多。  
有疑问时，选 `NEEDS_POC`，而不是 `REFUTED`。

---

## 陷阱 1：分布式控制（最常见陷阱）

**错误推理**："我看到了鉴权装饰器 / 中间件，所以这个接口是安全的。"

**危险所在**：
- 中间件可能只保护了部分路由，或通过条件跳过（`if not DEBUG`）
- 认证 ≠ 授权：`@login_required` 只证明"已登录"，不证明"有权限操作这个资源"
- 鉴权可能只检查了"用户存在"，没有检查资源归属（org_id / user_id / account_id）
- 同一个函数可能被多个入口调用，鉴权只保护了其中一条路径

**正确做法**：追踪鉴权代码的实现，确认它在**所有调用路径**上都检查了**正确的资源归属**，而不只是"已登录"。

---

## 陷阱 2：认证 ≠ 防重放

**错误推理**："这个接口有签名校验 / JWT 验证，所以不会被重放。"

**危险所在**：认证（你是谁）和防重放（这个请求只能用一次）是两个**独立**的控制，缺一不可：
- 签名验证只证明请求来自合法方，不防止该合法请求被截获后重放
- 必须同时有：nonce 消费记录（已用则拒）+ 时间戳有效窗口（≤ 5 分钟）

**正确做法**：找到 nonce 的存储位置，确认消费后被标记为已用且不可复用，且时间戳窗口有上限。

---

## 陷阱 3：只排除了部分调用路径

**错误推理**："从入口 A 触达不了这个漏洞位置，所以是误报。"

**危险所在**：同一个危险函数 / 数据路径可能被多个入口调用。只排除了已知的一条路径，不等于整个漏洞不存在。

**正确做法**：grep 函数名、逐层上溯找到**所有**调用该函数的位置，对每条路径都确认有防护。只有全部路径都有保护，才能 REFUTED。

---

## 陷阱 4：锁 / 事务的返回值被丢弃

**错误推理**："代码里调用了 `select_for_update()` / `acquire_lock()`，所以是线程安全的。"

**危险所在**：
- `select_for_update()` 的返回结果被丢弃，后续操作用的是没有锁的对象
- 锁在事务外获取，critical section 不在事务内
- Redis `setnx` 实现有缺陷，TTL 未设置，或锁未原子释放

**正确做法**：确认锁的**获取、持有、释放**的代码边界和被保护的 critical section 完全重叠。

---

## 陷阱 5：误导性输入来源

**错误推理**："`id` 来自 session / config，不是 request body，所以攻击者控制不了。"

**危险所在**：
- Session 本身可能被攻击者修改（session fixation、子域 cookie 注入）
- 某些 framework 会把 session 和 request 参数混合处理
- 存储在 DB 里后被再次读出的值，在写入时可能被攻击者控制（second-order injection）

**正确做法**：追踪到该值的**最初写入来源**，确认攻击者完全无法控制任何写入路径。

---

## 陷阱 6：可达性未完整验证

**错误推理**："这个接口在内部工具 / 内网服务里，攻击者访问不到。"

**危险所在**：
- "内部"路由可能被意外挂到了外部路由上（`include()` 了整个 blueprint）
- 内部服务信任了来自外部的特定 header（`X-Internal: true` 风格的伪认证）
- 攻击者可以通过 SSRF 从外部到达内部服务

**正确做法**：查路由注册代码，确认该入口在所有环境下都无法从外部直接访问，且不能被 SSRF 间接触达。

---

## 陷阱 7：影响小 ≠ 不成立

**错误推理**："这个漏洞影响不大，不值得上报。"

**危险所在**：impact 调整的是 severity，不是 verdict。一个低影响的真实漏洞，verdict 仍是 confirmed，只是 severity 应该是 LOW 或 MEDIUM。

**正确做法**：verdict 只看漏洞是否成立（攻击路径是否完整可达）。影响大小用 severity 表达，不用来判断 verdict。

---

## REFUTED 的正确使用条件

只有**以下所有条件同时满足**，才能使用 `REFUTED`：

1. 你读文件逐层上溯追踪了所有调用路径，**无一例外**都有完整防护
2. 防护在被保护代码实际执行**之前**已生效（不是在危险操作之后）
3. 防护检查的是正确的**资源归属**（org_id / user_id / account_id），而非只检查"已登录"
4. 你引用了具体的**代码行号**作为证据

任一条件不满足 → 选 `NEEDS_POC`，不选 `REFUTED`。
